ADR-0001 · 以「有凭证」定义可用模型并参与 list_models 过滤

Status: Accepted · Date: 2026-06-01 · 设计文档: list-models-usable-filter

Context

list_models / REST GET /api/settings/models 默认按 enabledModels 白名单过滤，不看凭证，列出无 API key 的模型，用户一选就在运行时报错（stopReason=error）。

Decision

「可用」定义为 hasCredentials()==true——凭证来源含 custom provider、模型内嵌 key、 auth.json、settings.provider.<id>.apiKey、provider 环境变量（含 Google ADC / AWS Bedrock 哨兵），详见设计文档 §2。默认 list_models 列表据此过滤。

考虑过的选项

A · 复用 hasCredentials（选中）
复用既有凭证分层，单一事实源，无重复逻辑
每模型一次 resolve，需 provider 级缓存才能完全消除 O(N) 文件读
B · 另写一套可用性判断
可为列表场景定制规则
与 resolver 双源，凭证语义易漂移、双重维护

Rationale

hasCredentials 已封装全部凭证来源，是 catalog 与运行时鉴权的同一判据；性能可用 provider 级缓存解决，不构成另起炉灶的理由。

Consequences

＋ 默认列表不再出现一选就失败的模型。
－ 无 key 环境默认列表可能仅剩 custom（见设计文档 E1），需 all:true 兜底。
